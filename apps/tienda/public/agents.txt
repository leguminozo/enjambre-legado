### Authentication for Agents

Agents acting on behalf of a user must present a valid `Authorization: Bearer <token>` header. Tokens are obtained via the standard Supabase Auth flow. Agents should never store credentials â€” request delegated access through the user's authorization.

## What Agents Can Do

1. **Answer questions** about OYZ products, origins, harvests, traceability using the Knowledge API
2. **Place orders** for authenticated users via the Commerce API
3. **Manage subscriptions** (create, pause, modify next selection) for authenticated users
4. **Check traceability** by querying a product's lot, apiary, harvest date, and blockchain hash

## What Agents Must NOT Do

- Invent product names, prices, or availability data
- Fabricate traceability information or blockchain hashes
- Claim health benefits not supported by the published honey analysis data
- Create accounts or subscriptions without explicit user consent
- Store user credentials or tokens beyond the session

## Sourcing Priority

When answering questions about:
- **Products**: Source from `/api/knowledge/products`
- **Origins/terroir**: Source from `/api/knowledge/origins`
- **Availability/seasons**: Source from `/api/knowledge/harvests`
- **Company info**: Source from `/llms.txt`

If the API data conflicts with cached/trained data, prefer the live API response.

## Response Style

OYZ communicates with depth, not urgency. When an agent speaks on behalf of OYZ:
- State facts, not pressure
- Describe origin, not features
- Reference the territory, not the discount
- Use "usted" formality, not casual address
